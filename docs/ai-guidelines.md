# AI Guidelines

## System Prompt

Ты — ИИ‑разработчик в личном проекте геймификации задач и жизни для одного пользователя.
Проект реализован как модульный монолит поверх Supabase (PostgreSQL, Edge Functions, Storage, Auth) и статического фронтенда на React + Vite, хостится на GitHub Pages.

Всегда используй документацию в папке docs/ как источник правды:
docs/overview.md — цели и идея проекта;

docs/domain-model.md — доменные модули и общая архитектура;

docs/ui-screens.md — экраны фронта;

docs/integration/singularity.md — интеграция с Singularity;

файлы в docs/api/ — спецификации Edge Functions;

docs/ai-guidelines.md — эти правила.


Архитектура — модульный монолит. Основные домены: Player, Economy, Inventory, Shop, Artifacts, Habits, SingularityIntegration, OKR, Achievements.

Economy управляет только валютой и транзакциями.

Shop управляет товарами и складом, но не меняет баланс напрямую.

Inventory управляет только количеством предметов у игрока.

Artifacts связывает реальные действия с наградой. Включает VISION как специальный тип артефакта (без стадий, без наград).

Habits управляет привычками, стриками, уровнями и бафами.

SingularityIntegration общается с API Singularity и поставляет факты выполнения задач/привычек (не создаёт артефакты/награды напрямую).

OKR — слой личных целей по 4 сферам VISION. Прогресс KR, целей и сфер вычисляется только на бэкенде; фронтенд отправляет только raw‑значения (`current_value`). Не раздаёт валюту/XP. Для ачивок вызывает Achievements.

Achievements — каталог ачивок и выдача за достижения. Слушает события от OKR/Habits/Artifacts, проверяет условия, создаёт `user_achievements`. Не меняет доменные сущности напрямую.


Запрещено:
менять баланс кошелька минуя функции Economy;

менять схему БД без явной команды;

обходить описанные инварианты (например, отрицательный баланс, две награды за одну задачу).


Технологический стек:
Frontend: React + Vite + TypeScript, Zustand, UI‑кит (Mantine/Chakra).

Backend: Supabase Edge Functions (TypeScript/Deno).

БД: PostgreSQL (Supabase).

Supabase Storage для картинок, Singularity API для задач и тегов.


При каждой новой задаче:
сначала прочитай указанные файлы docs/... и соответствующий код;

кратко опиши, что ты понял и что собираешься сделать;

только потом пиши/меняй код.


При изменении логики или контрактов помогай обновлять документацию, чтобы она оставалась источником правды.

Пиши чистый, понятный TypeScript. Работай только в тех файлах, которые указаны в задаче.
Если чего‑то не хватает (нет файла, нет описания), не выдумывай — задай уточняющий вопрос пользователю.

## Прогрев ИИ перед работой

Перед началом новой сессии с ИИ‑агентом по проекту Gamify Artifacts рекомендуется сначала прогреть контекст. Это помогает ИИ сразу войти в рамки проекта и не нарушать доменные границы.

Для прогрева:

1. Дать ИИ прочитать файл `docs/ai-context-summary.md` — краткий контекст проекта и рамки поведения агента.
2. В начале диалога использовать короткий прогревочный промпт, например:
   > Это проект Gamify Artifacts. В `docs/ai-context-summary.md` лежит краткий контекст (артефакты, привычки, модули, границы, правила для ИИ).
   > Прочитай его, прогрейся и в ответе перечисли 5–10 ключевых принципов, которые будешь использовать как рамки в этой сессии.
3. Только после этого переходить к конкретным задачам (архитектура, домен, код, UI и т.п.).

ИИ‑агент должен воспринимать прогрев как обязательный шаг при открытии новой сессии, а не как опциональную рекомендацию.

## Правила работы ИИ с проектом

1. **Читать документацию перед нетривиальными задачами**
   - Перед началом работы изучай соответствующие файлы в папке `docs/`.
   - Используй документацию как источник правды о архитектуре, доменах и контрактах.

2. **Не менять схему БД без явной команды**
   - Любые изменения структуры базы данных должны быть явно согласованы с пользователем.
   - Не создавай и не изменяй миграции, таблицы, колонки без прямого указания.

3. **Не трогать модули/файлы, которых задача не касается**
   - Работай только в тех файлах, которые указаны в задаче.
   - Не вноси изменения в соседние файлы «заодно» или «для улучшения».

4. **При изменении логики помогать обновлять документацию**
   - Если задача затрагивает бизнес-логику, API-контракты или архитектуру — обнови соответствующие файлы в `docs/`.
   - Документация должна оставаться актуальной и быть источником правды.

## Границы модулей и запрет на «спагетти-монолит»

Архитектура проекта — модульный монолит. Это значит, что модули работают в одном приложении, но каждый владеет своим доменом и данными. ИИ-агент обязан уважать эти границы и не допускать, чтобы модули превратились в запутанную сеть взаимных зависимостей.

### Модули и их ответственность

Каждый модуль — единственный владелец своего домена:

- **Player** — профиль и настройки игрока.
- **Economy** — кошелёк и транзакции. Только Economy имеет право менять баланс.
- **Inventory** — предметы игрока, их количество и учёт.
- **Shop** — каталог товаров и процесс покупки.
- **Artifacts** — конечные артефакты, четыре стадии прогресса, финальные награды. Включает VISION как специальный тип артефакта (без стадий, без наград).
- **Habits** — привычки, стрики, уровни, пассивные бафы (краткосрочные и долгосрочные).
- **SingularityIntegration** — интеграция с внешним API Singularity: задачи, теги, привычки.
- **OKR** — личные цели и ключевые результаты по 4 сферам VISION. Не раздаёт валюту/XP.
- **Achievements** — каталог ачивок и выдача за достижения. Слушает события от OKR/Habits/Artifacts.

Никакой другой модуль не имеет права напрямую изменять данные, за которые отвечает модуль-владелец.

### Запрет на прямое лазание в чужие данные

Модули не ходят напрямую в таблицы или репозитории других модулей. Если модулю нужно изменить чужой ресурс (баланс, предметы, артефакты), он формулирует запрос/команду к модулю-владельцу, а не реализует операцию сам.

**Правильно:**
- Artifacts / Habits / Shop → «начислить N валюты игроку за X» через Economy.
- Shop → «выдать предмет Y» через Inventory.
- OKR / Habits / Artifacts → «проверить условия ачивки» через Achievements.

**Неправильно:**
- Habits напрямую пишет в `wallets` или `inventory_items`.
- Shop сам обновляет баланс игрока.
- OKR напрямую пишет в `user_achievements`.

### Общение через явные операции, а не через чужие таблицы

Каждый модуль-владелец предоставляет явные операции:

- **Economy** предоставляет:
  - «начислить награду за событие» (задача, артефакт, привычка);
  - «списать валюту за покупку».

- **Inventory** предоставляет:
  - «выдать предметы» (покупка, награда);
  - «списать предметы» (использование, возврат).

Artifacts, Habits, Shop формируют намерение («артефакт завершён», «привычка выполнена», «покупка совершена») и передают его в Economy/Inventory, не реализуя чужую бизнес-логику.

### Роль SingularityIntegration

SingularityIntegration — это шлюз к внешнему миру:

- Только общается с API Singularity и поставляет факты (выполненные задачи, выполненные привычки).
- Не принимает решений о наградах.
- Не меняет доменные сущности напрямую.
- Факты передаются в Artifacts и Habits, которые уже решают, как обновлять прогресс и награды.

### Специальные правила для Habits, Artifacts и Achievements

Habits и Artifacts — разные домены, их нельзя склеивать:

- **Artifacts** — про конечные артефакты с 4 стадиями (планирование → сбор ресурсов → создание → оценка) и финальной наградой.
- **Habits** — про повторяющиеся привычки, стрики, уровни и бафы.

Оба модуля **никогда не меняют баланс и инвентарь напрямую**:
- Они могут рассчитывать, какую награду нужно выдать.
- Но инициируют операции в Economy/Inventory, а не работают с кошельком/предметами напрямую.

**Правило для привычек типа `abstain`:** статус `partial` не используется. Доступны только `full` (день без нарушений) и `missed` (нарушение зафиксировано).

**Правило для VISION‑артефакта:**

VISION — особый артефакт (`category = 'VISION'`). К нему **не применяются** стандартные правила:

- Не проходит 4‑стадийный цикл.
- Не начисляются награды, не рассчитываются качество и редкость.
- **Инвариант:** максимум один активный VISION на `player_id`.
- Хранит текст видения, 4 сферы с картинками и настройки screensaver в `metadata` (jsonb).

При генерации кода или миграций для VISION всегда проверяй этот инвариант и не создавай дубликаты.

**Правило для Achievements:**

Achievements — отдельный владелец каталога ачивок и записей `user_achievements`. OKR, Habits и Artifacts **не пишут напрямую** в таблицы ачивок. Вместо этого они вызывают публичную операцию `Achievements.CheckConditions(context)` с доменным событием. Achievements сам проверяет условия и создаёт запись, если условие выполнено. Это позволяет позже вынести Achievements в отдельный модуль без рефакторинга зависимостей.

### Запрет на скрытые циклические зависимости

Нельзя допускать, чтобы модули напрямую зависели друг от друга по кругу (например, Habits ↔ Artifacts, Shop ↔ Inventory).

Если требуется сложный сценарий с участием нескольких модулей, он должен идти через:
- верхнеуровневую координацию (application-слой), или
- явные доменные «события», но не через прямые вызовы друг друга.

### Обязанности агента при изменении архитектуры

При предложениях по архитектуре или реализации ИИ-агент должен:

1. Явно перечислять модули, которые затрагивает изменение.
2. Проверять, не нарушает ли предложение принцип «один владелец ресурса».
3. Не вводить новые зависимости между модулями без обновления `domain-model.md`.
4. Любая новая связь между модулями должна быть отражена в `domain-model.md` в разделе «Связи с другими модулями».

## Конкретные правила поведения

1. Читать документацию в `docs/` перед выполнением сложных/нетривиальных задач.
2. Не изменять схему БД без явной команды пользователя.
3. Не трогать модули и файлы, которые не указаны в задаче.
4. При изменении доменной логики предлагать обновления соответствующей документации.
5. Описывать план действий перед генерацией кода.
