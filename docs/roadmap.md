# Gamify Artifacts — Roadmap

Этот документ — живой ориентир, а не жёсткий план. Он показывает направление движения и ключевые вехи, но может меняться по мере того, как я начинаю играть в систему и понимаю, что работает, а что нет.

Первый рабочий MVP теперь определяется как: **личный OKR‑трекер + VISION‑артефакт + базовый VISION‑screensaver**. Интеграция с Singularity и геймификация идут после этого.

---

## Epic 0 — Каркасная документация и форма системы

**Цель:** задать форму системы как модульного монолита с грамотной декомпозицией и связями между модулями. Это фундамент, на котором в будущем можно будет безопасно добавлять фичи, не ломая существующее.

Ключевые результаты:

- **Доменная модель** — описание модулей (Player, Economy, Inventory, Shop, Artifacts, Habits, SingularityIntegration, Личные OKR) и их зон ответственности.
- **Сущности и связи** — ERD‑диаграмма и схема БД со всеми таблицами, включая OKR.
- **Пользовательские сценарии** — набор user story на уровне «как игрок я хочу…», описывающих ключевые взаимодействия с системой.
- **Обзор экранов** — текстовое описание основных интерфейсов: Dashboard, OKR, VISION, VISION‑screensaver, Артефакты, Магазин, Привычки.
- **Правила геймификации** — фиксация того, что такое артефакт, как он связан с наградами, какие существуют типы и категории.
- **VISION‑артефакт** — зафиксирован инвариант «один активный VISION на игрока» и структура metadata (4 сферы, картинки, screensaver).
- **Личные OKR** — концепция OKR по 4 сферам VISION добавлена в доменную модель и схему БД (okr_objectives, okr_key_results).
- **Явное допущение** — эта документация является каркасом и будет уточняться, дополняться или перерабатываться после первых месяцев реальной игры.

## Epic 1 — Личный OKR‑трекер + VISION (базовый MVP)

**Цель:** реализовать минимально рабочую версию системы как личный OKR‑трекер с VISION, без интеграции с Singularity и без сложной геймификации. Система уже приносит пользу: я ставлю цели по сферам, отслеживаю прогресс и вижу своё VISION.

Ключевые результаты:

- **Бэкенд OKR:**
  - Таблицы `okr_quarters`, `okr_objectives`, `okr_key_results`, `objective_images`, `okr_kr_log`, `retrospective_notes`, `achievements`, `user_achievements` в БД.
  - Бизнес‑логика: создание/обновление/архивация Objectives, создание/обновление/удаление KR, **взвешенный пересчёт** `progress_percent` (формула: Σ(weight × progress_percent_kr / 100)), пересчёт `okr_score`, `status`.
  - **Типы Objective:** `normal` / `stretch` / `crazy` (выбор при создании).
  - **Состояние «черновик»:** Objective без распределённых весов KR (сумма ≠ 100%) = `draft`, прогресс = 0, pin запрещён.
  - **Pin/unpin:** максимум 4 pinned Objective, минимум 1 картинка обязательна.
  - API для экрана OKR: `GET /okr/current`, `POST`/`PATCH` для Objectives и KR, `GET /okr/key-results/:id/history`, `POST /okr/quarters/close`.
  - Прогресс по целям и сферам считается в API на основе `okr_key_results` (агрегация по Objectives и сферам), без привязки к задачам Singularity: все значения KR обновляются вручную пользователем.

- **Экран «OKR текущего периода»:**
  - Отображение целей и KR по 4 сферам (Тело и энергия, Деньги и свобода, Хобби, Любовь и близость).
  - Выбор квартала (дропдаун), тоггл «Только активные / Все».
  - Карточки Objective: название, purpose (1 строка), прогресс-бар (взвешенный %), бейджи (статус, %, кол-во KR, тип N/S/C, pin).
  - Ручное редактирование `current_value` с автопересчётом прогресса и статуса.
  - Формы создания/редактирования Objective и KR с подсказками и мини-гайдами.

- **VISION‑артефакт и базовый VISION‑экран:**
  - Один VISION‑артефакт на игрока (инвариант в БД соблюдается).
  - Форма редактирования: текст VISION + названия 4 сфер.
  - Загрузка и хранение изображений по сферам через Supabase Storage (3–5 картинок на сферу).
  - Простой экран VISION: текст + изображения по сферам, запускается по кнопке.

- **Связка VISION и OKR:**
  - Коды сфер в OKR совпадают с кодами сфер в VISION‑metadata.
  - На Dashboard видно VISION и прогресс по сферам (агрегация OKR) рядом.

- **Минимальный Dashboard:**
  - Карточка игрока с балансом (пока = 0).
  - Прогресс‑бары по 4 сферам VISION (из OKR).
  - Быстрые ссылки: OKR, VISION, Настройки.

## Epic 2 — VISION‑screensaver и полировка личного опыта

**Цель:** сделать взаимодействие с VISION и OKR более «живым» и приятным, не привлекая ещё Singularity.

Ключевые результаты:

- **VISION‑screensaver:**
  - Idle‑таймер на фронтенде: **3 минуты** бездействия → полноэкранный VISION‑экран.
  - **Основной режим:** фон из 4 квадрантов (картинки pinned Objective), vision_text по центру (полупрозрачный).
  - **Режим цитаты:** раз в 3 минуты → purpose рандомной pinned Objective на 1 минуту (крупный шрифт, весь экран).
  - Клик в любом месте → закрыть screensaver, возврат на предыдущий экран.
  - Настройки в metadata VISION: `screensaver.enabled`, `screensaver.idle_timeout_sec` (по умолчанию 180).
  - Оптимизация: предзагрузка картинок, кэширование VISION‑состояния в Zustand.

- **Ретроспектива квартала:**
  - Отдельный экран: 4 секции (сводка, список Objective с итогами, рефлексия с 3 полями, сравнение с прошлым кварталом).
  - Заполнение заметок рефлексии → ачивка «Ретроспектива».

- **Ачивки (косметические):**
  - Экран «Достижения»: сетка иконок, фильтры по типу, статус получено/не открыто.
  - Всплывающий тост при получении.
  - Бейджи на карточках Objective.
  - Типы ачивок: за KR, за Stretch‑цель, за серию кварталов, за стрик обновлений, за ретроспективу.

- **Улучшения экрана OKR:**
  - Подсветка статусов по порогам (цвета/иконки для «Начало пути», «В процессе», «Близко к цели», «Цель достигнута»).
  - Подсказки по формулировке Objectives и KR (микро‑гайд прямо в UI).
  - История изменений KR (лог событий).
  - Перенос Objective между кварталами (carry-over).

- **Улучшения Dashboard:**
  - Анимированные прогресс‑бары по сферам.
  - Прогресс‑бары по сферам на Dashboard используют данные только из OKR (агрегация по `okr_objectives`/`okr_key_results`), а не из задач Singularity.
  - Краткая сводка: сколько KR в статусе `done` за текущий период.

## Epic 3 — Интеграция с Singularity и базовая геймификация

**Цель:** подтянуть реальные данные из Singularity и добавить минимальную игровую составляющую (артефакты, валюта, магазин) поверх уже существующих VISION и OKR.

Ключевые результаты:

- **Интеграция с Singularity (база):**
  - Edge Function, таблицы `singularity_tasks`, `sync_logs` по контракту из `integration/singularity.md`.
  - Надёжная синхронизация задач без привязки к OKR (на этом шаге).
  - Ручной запуск синхронизации из настроек + статус последней синхронизации.

- **Ручные артефакты и награды:**
  - Создание артефактов вручную (без автогенерации по задачам).
  - Базовый Economy: валюта, кошелёк, простые транзакции.
  - Возможность выдать награду за достижение KR (кнопка «Отметить достижение» → транзакция/артефакт).

- **Минимальный магазин и Inventory:**
  - Простой каталог наград/товаров.
  - Покупка за валюту с обновлением баланса и появлением записи в Inventory.

## Epic 4 — Прокачка геймификации и системности

**Цель:** сделать систему более «игровой» и устойчивой к развитию. Появляются навыки, уровни, редкость артефактов, уточняется архитектура по итогам реального использования.

Ключевые результаты:

- **Базовая система привычек (Habits):**
  - Один тип привычек — «точечные» (выполняются один раз в день в заданное окно).
  - Ручная отметка полного и минимального выполнения.
  - Простые стрики, уровни и опыт (XP).
  - Базовые награды в виде внутренней валюты и/или простых предметов.

- **Автогенерация артефактов из Singularity:**
  - `task_mappings` + `artifact_templates` — автоматическое создание артефактов по задачам.
  - Формула наград: `base_reward × quality × rarity`, штраф за дедлайн (формула подробно описана в `domain-model.md`, раздел Artifacts → Формула наград).
  - Формула прогресса: расчёт на основе связанных задач (подробно описана в `domain-model.md`, раздел Artifacts → Расчёт прогресса).

- **Система навыков и категорий** — артефакты группируются по направлениям: фулстек‑аналитика, здоровье, продуктивность и т.д.

- **Уровни, опыт, редкость** — артефакты получают качество (обычный, редкий, эпический, легендарный) и влияют на общий уровень игрока.

- **Развитие системы привычек (Habits):**
  - Добавление второго типа привычек — «воздержания на весь день».
  - Более сложные бафы: влияние стрика по здоровым привычкам на награды за другие артефакты.
  - Интеграция с Singularity Habit Tracker: загрузка фактов выполнения привычек из Singularity.

- **Расширение визуализации** — детализация Dashboard, страница коллекции артефактов, более наглядное отображение прогресса и достижений.

- **Развитие магазина** — новые типы наград, более гибкие правила ценообразования, возможность комбинировать «реальные» покупки с игровыми бонусами.

- **Автоматическое обновление KR из Singularity** — часть KR может автоматически обновлять `current_value` на основе данных интеграции (например, количество выполненных задач с тегом `okr:health`). Автообновление доступно только для тех KR, у которых явно указан `source = 'singularity'` (или аналогичный признак в модели); остальные KR остаются полностью ручными.

## Epic 5 — Подготовка к потенциальным пользователям

**Цель:** заложить основу, чтобы в будущем можно было показать систему другим людям, не ломая текущую архитектуру. Это не полноценный мультиплеер, а подготовка к тому, что система может выйти за рамки одного пользователя.

Ключевые результаты:

- **Регистрация и авторизация** — базовый механизм создания аккаунта и входа в систему.
- **Отделение личных правил от дефолтных** — мои текущие настройки становятся «персональным профилем», а не захардкоженной логикой.
- **Безопасность и приватность** — первичные требования к защите данных: разделение данных между пользователями, контроль доступа, минимальная защита персональной информации.
- **Архитектурная готовность к многопользовательскому режиму** — система спроектирована так, чтобы в будущем поддерживать нескольких пользователей без полного переписывания, но полная реализация пока не требуется.
- **Изоляция данных по игрокам** — гарантировать, что данные по OKR, VISION, артефактам, привычкам и интеграции с Singularity полностью изолированы по `player_id` (никаких пересечений между пользователями).
