# Use Cases — Модуль OKR

Детальные сценарии взаимодействия пользователя с системой. Используются разработчиками для реализации бизнес-логики и QA для написания интеграционных/端到端 тестов.

> **Как читать:** Каждый Use Case описывает поведение системы, а не пиксели UI. Ссылки на конкретные элементы интерфейса — в `ui-screens.md`. Ссылки на формулы и инварианты — в `domain-model.md`.

## UC-01: Создание Objective с Key Results

| Поле | Значение |
|------|----------|
| **ID** | UC-01 |
| **Название** | Создание Objective с Key Results |
| **Актор** | Игрок (Player) |
| **Предусловия** | 1. Игрок аутентифицирован.<br>2. Выбран активный квартал (`status != completed`).<br>3. Открыт экран OKR. |
| **Постусловия** | 1. Objective создана со статусом `draft`.<br>2. KR созданы с `weight = null`.<br>3. Objective отображается в списке с бейджем «Черновик». |

### Основной поток (Happy Path)
1. Игрок инициирует создание новой Objective.
2. Система открывает форму ввода (сфера, `title`, `purpose`, `objective_type`).
3. Игрок заполняет обязательные поля и подтверждает создание.
4. Система создаёт запись в `okr_objectives` (`status = draft`, `progress_percent = 0`).
5. Игрок инициирует добавление KR к созданной Objective.
6. Система открывает форму KR (`title`, `type`, `start_value`, `target_value`, `unit`).
7. Игрок заполняет поля и сохраняет KR.
8. Система создаёт запись в `okr_key_results` (`weight = null`).
9. Система обновляет UI, показывает Objective в статусе `draft`.

### Альтернативные потоки
- **3a. Валидация не прошла** (пустой `title`, неверная сфера):
  1. Система подсвечивает ошибки в форме.
  2. Создание Objective отменяется. Поток возвращается к шагу 2.
- **5a. Игрок отменяет добавление KR**:
  1. Система закрывает форму KR.
  2. Objective остаётся в статусе `draft` без KR. Поток завершается.

### Бизнес-правила
- **BR-1:** Objective без распределённых весов KR = `draft`. Прогресс = 0. Pin запрещён. *(domain-model.md, раздел «Состояние черновик»)*
- **BR-2:** `purpose` опционален, но рекомендуется для отображения в screensaver.
- **BR-3:** Допустимые типы KR: `numeric`, `binary`, `qualitative`. *(domain-model.md, раздел «Расчёт прогресса»)*

### Исключения
- **E-1: Ошибка сети/сервера (4xx/5xx)** при создании Objective или KR.
  - Система откатывает локальное состояние формы.
  - Показывает тост: «Не удалось сохранить. Проверьте соединение».
- **E-2: Квартал уже закрыт**.
  - Система блокирует форму, показывает предупреждение: «Квартал завершён. Создание целей недоступно».

### Связанные артефакты
- **UI:** Экран «Создание/редактирование Objective» (`ui-screens.md`)
- **API:** `POST /okr/objectives`, `POST /okr/key-results` (`okr-api.md`)

## UC-02: Распределение весов KR и активация Objective

| Поле | Значение |
|------|----------|
| **ID** | UC-02 |
| **Название** | Распределение весов KR и активация Objective |
| **Актор** | Игрок (Player) |
| **Предусловия** | 1. Objective в статусе `draft`.<br>2. У Objective есть ≥1 KR.<br>3. Сумма текущих весов KR ≠ 100%. |
| **Постусловия** | 1. Сумма весов KR = 100%.<br>2. Статус Objective изменён на `active`.<br>3. Прогресс Objective рассчитывается по взвешенной формуле.<br>4. Pin становится доступен. |

### Основной поток (Happy Path)
1. Игрок открывает интерфейс распределения весов для Objective в статусе `draft`.
2. Система отображает список KR с полями ввода веса и индикатором текущей суммы.
3. Игрок вводит веса для каждого KR (шаг 5%).
4. Система валидирует ввод в реальном времени:
   - Каждый вес ≥ 0 и кратен 5.
   - Сумма всех весов = 100%.
5. Игрок подтверждает применение весов.
6. Система обновляет `weight` у всех KR в БД.
7. Система меняет статус Objective на `active`.
8. Система пересчитывает `progress_objective` по формуле: `Σ(weight_kr × progress_percent_kr / 100)`.
9. Система обновляет UI: убирает бейдж «Черновик», активирует прогресс-бар и кнопку Pin.

### Альтернативные потоки
- **4a. Сумма весов ≠ 100%**:
  1. Система блокирует кнопку подтверждения.
  2. Показывает остаток/перебор (например, «Осталось распределить: 15%»).
- **4b. Вес не кратен 5%**:
  1. Система показывает ошибку валидации у конкретного поля.
  2. Предлагает авто-округление до ближайших 5% или ручной ввод.

### Бизнес-правила
- **BR-1:** Шаг веса = 5%. Сумма строго = 100%. *(domain-model.md, раздел «Взвешенный прогресс»)*
- **BR-2:** Переход `draft → active` происходит автоматически при выполнении BR-1.
- **BR-3:** До выполнения BR-1 прогресс Objective = 0, независимо от `current_value` KR.

### Исключения
- **E-1: Конфликт версий** (другое устройство изменило KR параллельно).
  - Система отклоняет сохранение (409 Conflict).
  - Предлагает перезагрузить данные Objective и повторить распределение.
- **E-2: Ошибка сервера** при обновлении весов.
  - Транзакция откатывается. UI возвращается к состоянию до подтверждения.

### Связанные артефакты
- **UI:** Экран «Создание/редактирование Key Result» (`ui-screens.md`)
- **API:** `PATCH /okr/key-results/:id` (многократный вызов или батч) (`okr-api.md`)

## UC-03: Закрытие квартала и фиксация результатов

| Поле | Значение |
|------|----------|
| **ID** | UC-03 |
| **Название** | Закрытие квартала и фиксация результатов |
| **Актор** | Игрок (Player) |
| **Предусловия** | 1. Квартал активен.<br>2. Есть ≥1 Objective в статусах `active` или `draft`. |
| **Постусловия** | 1. Все Objective квартала переведены в `completed`.<br>2. `result_status` заполнен для каждой Objective.<br>3. `is_pinned` сброшен в `false`.<br>4. Доступен экран ретроспективы. |

### Основной поток (Happy Path)
1. Игрок инициирует закрытие квартала.
2. Система показывает модалку подтверждения с итоговой сводкой (кол-во целей, средний %, распределение по сферам).
3. Игрок подтверждает действие.
4. Система фиксирует финальный `progress_percent` для всех Objective.
5. Система рассчитывает `result_status` для каждой Objective:
   - `success` (≥ 70%)
   - `partial` (40–69%)
   - `failed` (< 40%)
6. Система сбрасывает `is_pinned = false` для всех целей квартала.
7. Система создаёт записи в `okr_kr_log` с `event_type = 'quarter_closed'`.
8. Система запускает проверку условий ачивок (`Achievements.CheckConditions()`).
9. Система предлагает открыть экран ретроспективы.

### Альтернативные потоки
- **3a. Игрок отменяет действие**:
  1. Модалка закрывается.
  2. Квартал остаётся активным. Поток завершается.
- **5a. Objective типа `stretch` с результатом 60–80%**:
  1. Система помечает цель для проверки ачивки «Objective выполнен».
  2. `result_status` рассчитывается по общей шкале (порог `success` = 70%).

### Бизнес-правила
- **BR-1:** Закрытие квартала необратимо в MVP (нет операции reopen).
- **BR-2:** `result_status` и пороги ачивок — разные метрики. *(domain-model.md, раздел «result_status vs Stretch-ачивки»)*
- **BR-3:** KR блокируются от редактирования после закрытия (UI скрывает поля ввода).

### Исключения
- **E-1: Ошибка расчёта прогресса** (битые данные в KR).
  1. Система логирует ошибку, останавливает закрытие.
  2. Показывает: «Не удалось завершить квартал. Проверьте данные целей».
- **E-2: Потеря соединения** во время транзакции.
  1. Операция откатывается на уровне БД.
  2. UI показывает статус «Закрытие не завершено. Повторите попытку».

### Связанные артефакты
- **UI:** Экран «Ретроспектива квартала», модалка подтверждения (`ui-screens.md`)
- **API:** `POST /okr/quarters/close`, `GET /okr/retrospective` (`okr-api.md`)

## Глоссарий терминов

| Термин | Определение |
|--------|-------------|
| `draft` | Черновик Objective. Веса KR не распределены. Прогресс = 0. |
| `active` | Рабочая Objective. Веса KR = 100%. Прогресс считается. |
| `completed` | Квартал завершён. Финальный % зафиксирован. |
| `result_status` | Итоговая оценка цели: `success` / `partial` / `failed`. |
| `quarter_closed` | Событие в логе KR, фиксирующее момент закрытия квартала. |
