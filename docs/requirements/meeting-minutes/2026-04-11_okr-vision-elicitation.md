### Вопрос 10
**Тема:** KR: удаление, веса, история  
**Вопрос:** Сейчас в `db-schema.md` стоит `ON DELETE CASCADE` для KR при удалении Objective. Но ты сказал, что хочешь полный CRUD над KR. Если ты удаляешь KR — это влияет на прогресс Objective? Хочешь ли ты видеть историю изменений KR?

> **Оригинальный ответ PO:**  
> Я хочу видеть историю изменений (если это не сложно реализовать). Это будет полезно для рефлексии по итогам квартала. Если я удаляю Objective то логично что и key result его тоже удалятся, но я же могу изменить Objective и в таком случае key result не должны удаляться. Если я удаляю KR это должно влиять на прогресс Objective если в этом удаленном key result был прогресс, то он уберется из Objective. Я вижу такую формулу: у нас например 4 key result для одного objective, хотелось бы выставлять некие веса на конкретный key result. Если в одном месте поменялся вес, то это должно отражаться на остальных. Либо можно сделать проще. В момент создания key result можно в отдельном поле указывать сколько из общих 100% objective будет для этого key result. Например, я создаю 4 key result и вижу что 1 прям тяжелее выполнить чем остальные и даю ему 50% (т.е половину всего objective %), остается 50% на три остальных key result, можно сделать шаг в 5%, т.е минимально я могу дать одному key result 5%, максимально 100% (мало ли будет такой objective, где будет только 1 key result, маловероятно, но всё же), обычно это будет что то в стиле 25+25+25+25 если они равные по сложности или может типа такого 60+10+10+10. Сумма всех key result по весу прогресса должна быть ровно 100%. Надеюсь понятно объяснил и мне кажется это идеально удобно для использования и отображения прогресса в Objective (Например key result которому при создании я отвесил 50% при выполнении 10% прогресса этого key result его objective получит 5% прогресса, как только key result будет полностью выполнен, objective получит все 50% выданные key result при его создании). Отметь отдельно и прям подчеркни, что именно такую реализацию влияния прогресса key result на objective я хочу видеть в своем приложении. При удалении key result система должна предложить распределить освабодившиеся % прогресса между остальными key result или создать новый и дать ему этот вес, который остался после удаления key result. Так же очень важно, я хочу иметь возможность менять соотношения % прогресса между key result на протяжении всего квартала когда я достигаю objective, тк в начале при планировании я могу ошибиться с весами и не правильно распределить %

**Решение:** **КРИТИЧНО:** Изменить формулу прогресса со средней на взвешенную. Добавить `weight DECIMAL` в KR, сумма = 100%. Прогресс Objective = Σ(weight × progress_kr / 100). При удалении KR — предложить распределить %. Веса можно менять в течение квартала.  
**Статус:** ✅

### Вопрос 17
**Тема:** Награды за выполнение OKR — нужны ли?
**Вопрос:** В доменной модели есть экономика (валюта, магазин, артефакты). Хочешь ли ты, чтобы выполнение OKR давало награды? Например:
- **Вариант A:** За выполнение Objective → валюта (монеты). Чем выше % выполнения → тем больше монет.
- **Вариант B:** За выполнение Objective → XP (опыт) → уровень игрока.
- **Вариант C:** За выполнение Objective → уникальный артефакт (например, «Квартал покорителя»).
- **Вариант D:** OKR не даёт наград напрямую. OKR — это «сюжетные квесты», их награда — сам результат. Но можно дать бонус за 100% выполнение всех Objective в квартале.

> **Комментарий аналитика:** OKR — это про внутреннюю мотивацию («ради чего»). Внешние награды могут **подрывать** внутреннюю мотивацию (исследования по самоопределению). Но в геймификации награды — часть системы. Какой баланс ты хочешь?

> **Оригинальный ответ PO:**
> Не даём валюту / XP за Objective. OKR остаётся про «ради чего», а не про фарм. Деньги/XP — нет. Никакой валюты за Objective. Даём только косметику и ачивки. Хорошим результатом считаем 60–80% выполнения для целей уровня Stretch. Награда не за одно число (типа ровно 70%), а за попадание в диапазон и за серию кварталов: в этом квартале Stretch‑цель 60–80% → маленькая ачивка; несколько таких кварталов подряд → более редкая ачивка.

**Решение:** Только косметические ачивки, нет валюты/XP за OKR. Stretch-цели: 60–80% = ачивка. Серия кварталов = более редкая ачивка.
**Статус:** ✅

### Вопрос 26
**Тема:** Переходы статусов Objective
**Вопрос:** Какие статусы у Objective и как они меняются? Кто меняет: пользователь или система? Что происходит в конце квартала?

> **Оригинальный ответ PO:**
> Статусы: draft → active → completed → archived.
> draft: Objective без полных весов KR (сумма ≠ 100%).
> draft → active: автоматически, когда у всех KR веса = 100%. Можно дать кнопку «Активировать», но логика опирается на веса.
> active → completed: в конце квартала — все active → completed (с зафиксированным %). Не обязательно 100% — completed = «квартал завершён, подводим итог».
> completed → archived: пользователь или система архивирует старые Objective, чтобы не засоряли списки, но были доступны в ретроспективе.
> Если квартал закончился, а Objective не выполнен: цель идёт в completed, получает финальный %, статус результата в ретро: success / partial / failed. Перенос в новый квартал → новый Objective с parent_objective_id.

**Решение:** Статусы: draft → active → completed → archived. draft → active автоматически (веса = 100%). active → completed в конце квартала (система или кнопка «Закрыть квартал»). completed → archived (ручная или авто). Невыполненная цель = completed с result_status (success/partial/failed).
**Статус:** ✅
