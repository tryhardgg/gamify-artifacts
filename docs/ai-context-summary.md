## 1. Кто пользователь и что за проект

- Пользователь — опытный фулстек аналитик из финтеха, привыкший к продовому подходу.
- Проект **Gamify Artifacts** — личная игра‑система, превращающая реальные результаты и привычки в RPG.
- **Последовательность MVP:** 
  1. OKR‑трекер + VISION‑экран (базовый MVP)
  2. VISION‑screensaver + полировка UX
  3. Интеграция с Singularity + базовая геймификация (валюта, магазин, ручные артефакты)
  4. Прокачка геймификации (автогенерация, уровни, бафы)
  5. Подготовка к мультипользовательскому режиму.
  ⚠️ **Не предлагай фичи из эпиков 3–5, пока не закрыты 1–2.**

## 2. Ключевая доменная идея: артефакты

- **Артефакт** = осязаемый результат реального действия + 4 стадии + оценка + награда.
- **Жизненный цикл:** Планирование → Сбор ресурсов → Создание → Оценка. Не должен ломаться.
- **VISION‑артефакт** (`category = 'VISION'`):
  - Максимум 1 активный на игрока (инвариант БД: partial unique index).
  - НЕ проходит 4 стадии, НЕ имеет наград/качества/редкости.
  - Хранит в `metadata` (jsonb): `vision_text`, 4 сферы с картинками, `screensaver` настройки.
  - Используется для полноэкранного screensaver после N минут бездействия.

## 3. OKR (Личные цели)

- Отдельный слой над VISION, не модуль в строгом смысле.
- Состоит из `Objectives` (цели по 4 сферам) и `Key Results` (измеримые метрики).
- **Расчёт прогресса:** Выполняется ТОЛЬКО на бэкенде. Фронтенд шлёт только `current_value`.
  - `numeric`: `((current - start) / NULLIF(target - start, 0)) × 100`
  - `binary`: 0% до выполнения → 100% после.
  - `qualitative`: вводится вручную в UI (0–100%).
- **Агрегация:** Прогресс цели = среднее `progress_percent` её KR. Прогресс сферы = среднее прогресса активных целей в сфере. Все типы KR участвуют на равных правах.
- **На Dashboard:** Прогресс сфер отражает *текущий* агрегированный % за период. НЕ использует логику `progress_today` (она только для артефактов).
- **Удаление:** `ON DELETE CASCADE` по `objective_id`. Удаление цели стирает все её KR.

## 4. Привычки (Habits)

- Типы: `discrete` (точечные, окно времени) и `abstain` (воздержания на весь день).
- **Правило для `abstain`:** Статус `partial` НЕ используется. Доступны только `full` (день без нарушений) и `missed` (нарушение).
- Стрики, XP, уровни, бафы. Награды инициируются в Habits, но фактически меняют баланс/инвентарь через Economy/Inventory.
- Пропущенные дни: в MVP клиент создаёт `missed` записи при входе. В Epic 2 — server-side cron.

## 5. Архитектура: модульный монолит и границы

- **Player** — профиль, настройки, связь с Supabase Auth.
- **Economy** — валюта, кошелёк, транзакции. ЕДИНСТВЕННЫЙ владелец баланса. Инвариант: `balance >= 0`.
- **Inventory** — предметы игрока. ЕДИНСТВЕННЫЙ владелец количества.
- **Shop** — каталог, склад, покупка. Запрашивает списание через Economy, выдачу через Inventory.
- **Artifacts** — 4 стадии, прогресс, качество, редкость, инициация наград.
- **Habits** — привычки, XP, стрики, бафы, инициация наград.
- **SingularityIntegration** — поставляет факты из API. Не решает, какие награды выдавать.
- **OKR** — цели, KR, агрегация прогресса сфер. Не раздаёт награды.

**Правило:** Модули не лезут в чужие таблицы. Любое изменение чужого ресурса → через явную доменную операцию модуля‑владельца.

## 6. Критичные технические правила и инварианты

- **Миграции БД:**
  - `transactions.type` → обязателен `CHECK` или `ENUM` (`reward`, `purchase`, `habit_reward`, `artifact_reward`, `bonus`, `correction`, `refund`).
  - `okr_key_results.objective_id` → `ON DELETE CASCADE`.
  - `artifacts` → partial unique index: `UNIQUE (player_id) WHERE category = 'VISION' AND lifecycle_status = 'active'`.
- **Singularity API:** Задача считается «новой» (первое появление), если её `external_id` отсутствует в `singularity_tasks` для данного `player_id`. Правила `task_mappings` применяются только один раз. Изменение тегов после создания артефакта не триггерит пересоздание.
- **Кошелёк:** Создаётся автоматически при регистрации игрока. Начальный `balance = 0`. Стартовый бонус → отдельная транзакция `type='bonus'`, `source_module='system'`.
- **Конкурентные обновления OKR (MVP):** Не реализуются. Правило: «последнее сохранение побеждает».

1. **Не ломать доменную модель** и не смешивать логику модулей ради упрощения.
2. **Соблюдать границы:** один владелец ресурса, публичные операции, нет циклических зависимостей.
3. **Опираться на `docs/`:** `domain-model.md`, `db-schema.md`, `ui-screens.md`, `roadmap.md` — источники правды.
4. **Не менять стек** без запроса. Решения должны укладываться в объём «вечер/выходной».
5. **Не выдумывать сущности:** Если чего-то нет в схеме или домене — спроси. Не добавляй `PlayerStats`, `Notifications` и т.п. без согласования.
