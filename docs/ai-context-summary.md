# AI‑Context Summary для Gamify Artifacts

Этот файл — краткий контекст проекта и рамки поведения для ИИ‑агентов.  
Перед началом работы с проектом ИИ должен прочитать этот файл и использовать его как основу для ответов.

---

## 1. Кто пользователь и что за проект

- Пользователь — опытный фулстек аналитик из финтеха, привыкший к продовому подходу (эпики, доменная модель, аккуратная документация).
- Проект **Gamify Artifacts** — личная игра‑система, которая превращает реальную жизнь и работу аналитика в RPG:
  - реальные результаты (артефакты) и привычки (Habits) → награды, уровни, бафы.
- Цели:
  - сначала играть самому (личный MVP);
  - постепенно довести систему до состояния, пригодного для потенциальных пользователей.

---

## 2. Ключевая доменная идея: артефакты

- **Артефакт** — осязаемый результат реального действия, который можно:
  - сохранить (файл, запись, фото, скриншот, трекер),
  - показать,
  - использовать как основу для награды и прокачки.
- Не каждое действие — артефакт: если нет фиксируемого результата, это не артефакт.
- Примеры рабочих артефактов:
  - диаграммы (UML, BPMN, ERD, C4),
  - API спецификации (OpenAPI),
  - BRD/PRD, use cases, user stories,
  - архитектурные схемы и другие документы аналитика.
- Примеры жизненных артефактов:
  - запись о сне,
  - тренировка,
  - чек‑лист уборки,
  - скриншот шагомера,
  - запись о выполненной привычке.
- Артефакт = «трофей» реального мира, который система превращает в награду.

### 2.1. Жизненный цикл артефакта

Жизненный цикл артефакта всегда состоит из 4 стадий:

1. **Планирование**
2. **Сбор ресурсов**
3. **Создание артефакта**
4. **Оценка артефакта**

На стадии оценки:

- пользователь заполняет критерии;
- система считает качество и редкость (от «испорченного» до «мифического»);
- на основе этого выдаётся награда (валюта, опыт, предметы).

Этот 4‑стадийный цикл — фундаментальная часть домена и **не должен ломаться**.

## 3. Привычки как пассивные бафы (модуль Habits)

В проекте есть отдельный слой геймификации — **привычки** (Habits), которые дают пассивные и активные бафы.

### 3.1. Что такое привычка

- Привычка — повторяющееся действие или ограничение, которое игрок делает каждый день (или почти каждый) для улучшения себя.
- Работает как daily quest:
  - выполняется 0 или 1 раз в день (для точечных привычек),
  - имеет окно дня (например, 8:00–9:00),
  - даёт опыт, уровни и награды.

Типы привычек:

- **Точечные** — выполняются один раз в день в заданном окне (зарядка, медитация, чтение).
- **Воздержания** — ограничения на весь день (не ел сладкое, не залипал в рилсы и т.п.).

### 3.2. Уровни, XP и стрик

У каждой привычки есть:

- **Уровень** — растёт с накоплением опыта.
- **Опыт (XP)**:
  - полный день выполнения = 1 XP;
  - минимальное выполнение (лайт‑вариант) = 0.5 XP (не срывает стрик).
- **Стрик** — количество дней подряд, когда привычка выполнялась хотя бы минимально:
  - день выполнен (полностью или минимально) → стрик растёт;
  - день пропущен → стрик сбрасывается.

Порог XP по уровням можно настраивать (например, 0–5–10–15–20–…).

### 3.3. Награды и бафы

Привычки дают:

- **Моментальные награды**:
  - внутренняя валюта,
  - предметы или их части (например, часть порции мате).
- **Краткосрочные бафы** (на текущий день):
  - настроение, бодрость, снижение напряжения и т.п.
- **Долгосрочные бафы** (накопительный эффект):
  - здоровье, координация, осанка, уменьшение болей и т.п.

В домене:

- модуль Habits решает, какие награды выдать (валюта, предметы, обновление XP и стрика);
- модуль Economy изменяет баланс;
- модуль Inventory выдаёт предметы.

## 4. Архитектура: модульный монолит и доменные модули

Архитектура проекта — **модульный монолит** с чётко разделёнными доменными модулями.

Основные модули:

- **Player**
  - профиль игрока, базовые настройки, связь с Supabase Auth;
  - не управляет балансом, предметами и артефактами напрямую.

- **Economy**
  - валюта, кошелёк, транзакции;
  - единственный модуль, который имеет право менять баланс;
  - отвечает за инвариант «баланс не уходит в минус».

- **Inventory**
  - хранение предметов игрока;
  - выдача и расход предметов;
  - не управляет каталогом товаров и балансом.

- **Shop**
  - каталог товаров (реальные и игровые ништяки);
  - наличие/склад;
  - процесс покупки: проверка наличия, запрос списания валюты через Economy, выдача предметов через Inventory.

- **Artifacts**
  - жизненный цикл артефактов (4 стадии);
  - прогресс, дедлайны, качество, редкость, награды;
  - на завершении артефакта инициирует:
    - начисление валюты (через Economy),
    - выдачу предметов (через Inventory).

- **Habits**
  - привычки, XP, уровни, стрики, бафы;
  - считает награды за привычки;
  - не меняет баланс и инвентарь напрямую, только инициирует операции в Economy/Inventory.

- **SingularityIntegration**
  - интеграция с таск‑менеджером Singularity;
  - получает задачи/привычки/статусы;
  - поставляет внутренние факты (выполненные задачи и привычки) в Artifacts и Habits;
  - не принимает решений о наградах, только доставляет данные.

## 5. Жёсткие границы модулей (анти‑спагетти‑монолит)

Ключевой принцип: **у каждого типа ресурса есть один владелец, и только он меняет свои данные**.

Примеры владельцев:

- баланс и транзакции — только Economy;
- предметы — только Inventory;
- артефакты и их прогресс — только Artifacts;
- привычки, XP, уровни, стрики — только Habits;
- внешние данные Singularity — только SingularityIntegration.

Правила:

- Модули **не лезут** в чужие таблицы/репозитории.
- Любое изменение чужого ресурса идёт через явную доменную операцию модуля‑владельца.
  - Правильно: Artifacts/Habits/Shop вызывают операции Economy и Inventory.
  - Неправильно: эти модули сами создают записи транзакций или инвентаря в БД.
- SingularityIntegration:
  - только поставляет факты (выполненные задачи/привычки);
  - не создаёт артефакты, не меняет баланс, не выдаёт предметы.

---

## 6. Как модули общаются в коде

Рекомендации для реализации:

- Каждый модуль предоставляет ограниченный набор доменных функций/сервисов (внутренний API).
- Другие модули:
  - импортируют и вызывают эти функции,
  - не обращаются напрямую к чужим таблицам БД.

Допустимые паттерны:

- Прямые вызовы функций модуля‑владельца (подход по умолчанию для MVP).
- Тонкий application‑слой, который оркестрирует сложные сценарии (например, завершение артефакта вызывает Economy + Inventory).
- В будущем — внутренние доменные события (ArtifactCompleted, HabitCompleted) и подписчики, но это не обязательно для первой версии.

## 7. Правила поведения ИИ‑агента

При работе с проектом ИИ‑агент должен:

1. **Не ломать доменную модель**
   - Всегда помнить, что артефакт = реальный, фиксируемый результат + 4 стадии + оценка + награда.
   - Не смешивать логики Habits и Artifacts в один модуль «ради упрощения».

2. **Соблюдать модульные границы**
   - Не менять ответственность модулей без явного запроса пользователя и чёткого обоснования.
   - Не давать модулям прямой доступ к чужим данным; использовать только публичные операции модуля‑владельца.
   - Избегать циклических зависимостей между модулями.

3. **Опираясь на документацию `docs/`**
   - Считать `docs/overview.md`, `docs/domain-model.md`, `docs/ui-screens.md`, `docs/roadmap.md`, `docs/ai-guidelines.md` источниками правды.
   - При предложении архитектурных изменений явно указывать, какие файлы документации нужно обновить (особенно `domain-model.md`).

4. **Не менять стек без запроса**
   - Использовать фронтенд‑стек: React + TypeScript + Vite + Zustand + Mantine/Chakra.
   - Не предлагать кардинальную смену стека (Next, Vue и т.п.) без прямого запроса пользователя.

5. **Думать как помощник соло‑инди**
   - Предлагать решения, которые можно реализовать одному человеку в формате «вечер/выходной».
   - Отдавать приоритет простым и завершабельным механикам, а не теоретически идеальным, но тяжёлым.
