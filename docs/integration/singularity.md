# Интеграция с Singularity API

Этот документ описывает контракт взаимодействия с внешним API Singularity — таск-менеджером, который является источником фактов о выполненных задачах и привычках для системы Gamify Artifacts.

---

## 1. Обзор

- **Базовый URL:** `https://api.singularity-app.ru`
- **Версия API:** v2
- **Аутентификация:** Bearer token в заголовке `Authorization: Bearer <token>`
- **Токен** хранится только в Supabase secrets (`SINGULARITY_API_TOKEN`), используется исключительно в Edge Functions.
- **Фронтенд никогда не обращается напрямую к Singularity API.**

### player_id и multi-account

- **MVP:** один реальный пользователь = один `player` = один токен Singularity.
- **Масштабирование:** система спроектирована под нескольких игроков. Каждый `player` хранит свой токен Singularity (или ссылку на него). Edge Function запускается с контекстом `player_id`, и все записи в `singularity_tasks`, `task_mappings`, `sync_logs`, `singularity_habits` всегда пишутся с этим `player_id`. Данные разных игроков не пересекаются.

### Ограничения API

- API поддерживает только CRUD-операции — **нет вебхуков и подписок на изменения**.
- Синхронизация выполняется по расписанию (polling) или по ручному запуску.
- Повторяющиеся задачи создать через API нельзя.
- Совместные проекты не возвращаются через API (только личные).

## 2. Эндпоинты, используемые Gamify Artifacts

Из полного API Singularity нас интересуют только следующие ресурсы:

### 2.1. Задачи — `GET /v2/task`

**Зачем:** получение списка задач для синхронизации в `singularity_tasks`. Определение выполненных задач для обновления прогресса артефактов.

**Метод:** `GET`

**Параметры запроса:**

| Параметр | Тип | Обязательное | Описание |
|----------|-----|--------------|----------|
| `includeRemoved` | boolean | Нет | Включить удалённые задачи (по умолчанию `false`) |
| `includeArchived` | boolean | Нет | Включить архивные задачи (по умолчанию `false`) |
| `maxCount` | number | Нет | Ограничение количества задач |
| `includeAllRecurrenceInstances` | boolean | Нет | Показывать все экземпляры повторяющихся задач |
| `offset` | number | Нет | Смещение для пагинации |
| `projectId` | string | Нет | ID проекта для фильтрации |
| `parent` | string | Нет | ID родительской задачи (для подзадач) |
| `startDateFrom` | string (ISO) | Нет | Начальная дата диапазона |
| `startDateTo` | string (ISO) | Нет | Конечная дата диапазона |

**Ответ (ключевые поля задачи):**

| Поле | Тип | Описание | Маппинг на нашу систему |
|------|-----|----------|------------------------|
| `id` | string | Уникальный ID задачи | `singularity_tasks.external_id` |
| `title` | string | Название задачи | `singularity_tasks.title` |
| `note` | string | Описание/заметка | `singularity_tasks.description` |
| `start` | string (ISO) | Дата начала | — |
| `priority` | number | 0 — высокий, 1 — средний, 2 — низкий | — |
| `isNote` | boolean | `true` = заметка, `false` = задача | — |
| `journalDate` | string (ISO) | Дата архивации | → `status = 'archived'` |
| `deleteDate` | string (ISO) | Дата удаления | → `status = 'archived'` |
| `tags` | array | Теги задачи (если возвращаются) | `singularity_tasks.tags` |
| `status` | string | Статус задачи (если возвращается) | `singularity_tasks.status` |

**Критерий выполненности задачи:**

Определение статуса `singularity_tasks.status` из полей ответа Singularity API выполняется по следующему приоритету (сверху вниз, первое совпадение wins):

| Условие в ответе Singularity | `singularity_tasks.status` | Пояснение |
|------------------------------|---------------------------|-----------|
| Есть поле `isDone = true` или `status = 'done'` или аналогичное поле выполненности | `completed` | Задача выполнена, зачитывается в прогресс артефакта |
| `journalDate` установлен | `archived` | Задача заархивирована в журнале; считается завершённой и зачитывается в прогресс |
| `deleteDate` установлен | `archived` | Задача удалена в корзину; связь `artifact_task_links` сохраняется |
| Ни одно из вышеперечисленных | `open` | Задача в работе, не зачитывается в прогресс |

> **Важно:** на текущий момент API Singularity v2 не возвращает явного поля `isDone`/`doneDate`. При получении реальных ответов API необходимо проверить наличие такого поля. Если оно обнаружится — использовать его как основной критерий `completed`. Пока `journalDate` служит прокси для «задача завершена». В текущем MVP мы не используем отдельное поле `completed` из Singularity (если оно отсутствует или недоступно через API). Статус `archived` (по `journalDate`/`deleteDate`) служит прокси для «завершённых и убранных» задач. При появлении явного признака `done`/`isDone` в API логика может быть уточнена.

**Стратегия синхронизации:**

1. Edge Function вызывает `GET /v2/task` с `includeArchived=false`, `includeRemoved=false`.
2. Для каждой задачи:
   - если `journalDate` или `deleteDate` установлены → `status = 'archived'`;
   - иначе → `status = 'open'`.
3. Записи сохраняются/обновляются в `singularity_tasks`. **Архивированные задачи не удаляются** из `singularity_tasks` — запись сохраняется со `status = 'archived'`, чтобы `artifact_task_links` не «висяли на пустоту».
4. Для новых архивированных (завершённых) задач запускается процесс автогенерации артефактов через `task_mappings`.

**Пагинация и лимиты:**

- Рекомендуемое значение `maxCount`: **200** задач за один запрос.
- Если задач больше — использовать `offset` для постраничной загрузки (0, 200, 400, …).
- При изменении набора задач между страницами (задача создана/удалена во время пагинации) — живём с eventual consistency: следующая синхронизация подхватит изменения. Для MVP это приемлемо.
- Если API не вернёт `tags` (поле отсутствует или пустое): автогенерация артефакта по `task_mappings.tag_filter` **не срабатывает**. Задача сохраняется в `singularity_tasks`, но артефакт создаётся только вручную. В будущем как fallback можно использовать `projectId` для маппинга.

## 4. Обработка ошибок

### HTTP-коды

| Код | Значение | Действие |
|-----|----------|----------|
| `200` | Успех | Обработать ответ |
| `401` | Неавторизован | Проверить токен, залогировать ошибку в `sync_logs` |
| `403` | Доступ запрещён | Проверить права токена, залогировать |
| `404` | Не найдено | Задача/тег удалён, обновить статус в `singularity_tasks` |
| `429` | Rate limit | Повторить через N секунд (exponential backoff) |
| `500` | Ошибка сервера | Повторить через N секунд, залогировать в `sync_logs` |

### Retry-логика

- Максимум 3 повторных попытки.
- Exponential backoff: 1с → 2с → 4с.
- После исчерпания попыток — записать ошибку в `sync_logs.errors`.

## 5. Маппинг на нашу систему

### 5.1. Задачи → `singularity_tasks`

```
Singularity task          →  singularity_tasks
────────────────────────     ─────────────────────
id                        →  external_id
title                     →  title
note                      →  description
journalDate / deleteDate  →  status = 'archived'
открытая задача           →  status = 'open'
tags (если есть)          →  tags
полный ответ API          →  raw_data (исходный JSON без модификаций, тип jsonb;
                                 используется для отладки и постепенного расширения маппинга)
```

### 5.2. Теги → `task_mappings`

```
Singularity tag           →  task_mappings
────────────────────────     ─────────────────────
id                        →  tag_filter (массив)
title                     →  (отображение в UI)
```

### 5.3. Привычки → `singularity_habits` (на будущее)

```
Singularity habit         →  singularity_habits
────────────────────────     ─────────────────────
id                        →  external_id
title                     →  name
полный ответ API          →  raw_data (исходный JSON без модификаций, тип jsonb)
```

## 6. Сценарий синхронизации задач

### Инкрементальная синхронизация (по умолчанию)

```
1. Edge Function запускается (по расписанию или вручную) с контекстом player_id
2. Определить диапазон дат:
   a. Найти последнюю запись в sync_logs для этого player_id со status = 'success'
   b. Если есть → startDateFrom = sync_logs.finished_at (последняя успешная синхронизация)
   c. Если нет → startDateFrom = сегодня - 30 дней (дефолтный диапазон)
   d. startDateTo = сейчас
3. GET /v2/task?startDateFrom=...&startDateTo=...&maxCount=200 → список задач
   a. Если задач >= maxCount → использовать offset (0, 200, 400, …) для постраничной загрузки
4. Для каждой задачи:
   a. Найти существующую запись в singularity_tasks по (player_id, external_id)
   b. Если нет → INSERT
   c. Если есть → UPDATE (если изменились поля)
   d. Определить статус:
      - journalDate/deleteDate установлены → 'archived'
      - иначе → 'open'
5. Для новых архивированных (завершённых) задач:
   a. Найти matching task_mappings по tag_filter
   b. Если match найден → вызвать Artifacts.autoCreateArtifactsFromTemplate()
6. Записать результат в sync_logs:
   - player_id, started_at, finished_at, tasks_fetched, tasks_processed, status, errors
```

### Полный ресинк (forceFullSync)

Для отладки или восстановления данных после сбоя — ручной запуск с флагом `forceFullSync`:

```
1. Edge Function запускается с параметром forceFullSync = true
2. Диапазон дат не используется (запрос без startDateFrom/startDateTo)
3. GET /v2/task?includeArchived=true&includeRemoved=false&maxCount=200
4. Все остальные шаги — как в инкрементальной синхронизации
5. В sync_logs записывается status = 'success' с пометкой о полном ресинке
```

> **Примечание:** полный ресинк может занять несколько страниц пагинации. Каждая страница обрабатывается последовательно, результат фиксируется в одной записи `sync_logs`.

**Поведение task_mappings при изменении тегов:** правила `task_mappings` и `artifact_templates` применяются только в момент первого появления задачи (или первой синхронизации задачи, которая ещё не была обработана для автогенерации артефактов). Изменение тегов или других свойств задачи в Singularity после создания артефакта **не приводит** к пересозданию, удалению или изменению уже созданных артефактов. Это решение принято для сохранения стабильности и предсказуемости прогресса.

**Критерий «первого появления»:** задача считается новой, если её `external_id` отсутствует в таблице `singularity_tasks` для данного `player_id`. При первом появлении на задачу применяются правила `artifact_templates.rule`; при последующих синхронизациях для той же пары (`player_id`, `external_id`) новые артефакты по этим правилам уже не создаются.

## 7. Безопасность

- Токен `SINGULARITY_API_TOKEN` хранится **только** в Supabase secrets.
- Edge Function читает токен через `Deno.env.get('SINGULARITY_API_TOKEN')`.
- Токен **никогда** не передаётся во фронтенд.
- Токен **никогда** не логируется.
- При компрометации — немедленно отозвать и перегенерировать.

## 8. Связь с доменными модулями

**SingularityIntegration** владеет таблицами: `singularity_tasks`, `task_mappings`, `sync_logs`, `singularity_habits`.

- Только SingularityIntegration обращается к API Singularity и записывает данные в эти таблицы.
- Все остальные модули (**Artifacts**, **Habits**, **Economy**, **Inventory**, **Shop**) **не обращаются к Singularity API напрямую** и не читают таблицы SingularityIntegration напрямую. Они работают только через:
  - **Данные таблиц** — например, Artifacts читает `singularity_tasks` через `artifact_task_links` (junction-таблица, которой владеет Artifacts).
  - **Публичные операции модуля SingularityIntegration** — например, `getCompletedTasks(player_id, sinceDate)`, `getTaskById(external_id)`.

Это согласуется с принципом модульного монолита: «один владелец ресурса — один модуль, который меняет свои данные».

## 9. Расчёт прогресса артефактов

Данные из `singularity_tasks` и `artifact_task_links` используются модулем **Artifacts** для расчёта `artifacts.progress_total` и `artifacts.progress_today`.

Полное описание формул, алгоритмов и правил пересчёта — в `domain-model.md`, раздел **Artifacts → Расчёт прогресса артефактов**.

Кратко:
- Прогресс = `(задачи со status = 'archived' / все связанные задачи) × 100`.
- На старте все задачи равновесны (вес = 1).
- Пересчёт идемпотентный, при каждой синхронизации.

## 10. Прогресс сфер VISION: OKR vs Singularity

Прогресс по сферам VISION (`state`, `finances`, `hobbies`, `relationships`) **не рассчитывается напрямую по задачам Singularity**. Вместо этого:

- **Источник прогресса по сферам** — модуль **Личные OKR** (`okr_objectives` + `okr_key_results`). Игрок вручную задаёт цели и обновляет текущие значения KR.
- **Singularity** поставляет факты о выполненных задачах, которые используются для прогресса **артефактов**, но не для прогресса сфер VISION.
- **В будущем** (Epic 2): часть KR может автоматически обновлять `current_value` на основе данных из Singularity (например, количество выполненных задач с тегом `okr:health` автоматически обновляет KR в сфере `state`). Но это расширение, а не MVP.
