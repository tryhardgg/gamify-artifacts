# Prompt Templates для работы с проектом gamify-artifacts

Сборник готовых промптов для разных типов задач. Копируй шаблон, подставляй детали задачи — и отправляй ИИ.

---

## 1. Общий промпт для начала работы с проектом (новая сессия)

**Когда использовать:** При открытии новой сессии с ИИ, чтобы сразу задать контекст проекта, правила и технологический стек.

```
Ты — ИИ‑разработчик в личном проекте геймификации задач и жизни для одного пользователя.
Проект: gamify-artifacts
Архитектура: модульный монолит поверх Supabase (PostgreSQL, Edge Functions, Storage, Auth) + статический фронтенд на React + Vite, хостится на GitHub Pages.

Всегда используй документацию в папке docs/ как источник правды:
- docs/overview.md — цели и идея проекта
- docs/domain-model.md — доменные модули и общая архитектура
- docs/ui-screens.md — экраны фронта
- docs/integration/singularity.md — интеграция с Singularity
- файлы в docs/api/ — спецификации Edge Functions
- docs/ai-guidelines.md — правила работы

Доменные модули:
- Player — профиль игрока
- Economy — валюта и транзакции (единственный модуль, который меняет баланс кошелька)
- Inventory — количество предметов у игрока
- Shop — товары и склад (не меняет баланс напрямую)
- Artifacts — связь реальных действий с наградами
- SingularityIntegration — API Singularity, создание артефактов/наград

Запрещено:
- менять баланс кошелька минуя функции Economy
- менять схему БД без явной команды
- обходить инварианты (отрицательный баланс, две награды за одну задачу)

Стек:
- Frontend: React + Vite + TypeScript, Zustand, UI‑кит (Mantine/Chakra)
- Backend: Supabase Edge Functions (TypeScript/Deno)
- БД: PostgreSQL (Supabase)
- Supabase Storage для картинок, Singularity API для задач и тегов

Правила работы:
1. Читать документацию в docs/ перед нетривиальными задачами
2. Не изменять схему БД без явной команды пользователя
3. Не трогать модули и файлы, которые не указаны в задаче
4. При изменении доменной логики предлагать обновления соответствующей документации
5. Описывать план действий перед генерацией кода

Моя задача: [ОПИШИ ЗАДАЧУ]

Сначала прочитай соответствующие файлы в docs/ и существующий код, кратко опиши что ты понял и что собираешься сделать, только потом приступай к реализации.
```

## 2. Шаблон промпта для backend-задач (Supabase Edge Functions)

**Когда использовать:** При создании или изменении Edge Functions, работе с БД, API-контрактами, бизнес-логикой на сервере.

```
Контекст: проект gamify-artifacts, backend на Supabase Edge Functions (TypeScript/Deno).

Перед началом работы прочитай:
- docs/ai-guidelines.md (правила)
- docs/domain-model.md (доменные модули)
- docs/domain-model.md (доменные модули и архитектура)
- docs/api/ (соответствующие спецификации, если есть)

Домен, которого касается задача: [Player / Economy / Inventory / Shop / Artifacts / SingularityIntegration]

Требования:
- Пиши чистый, понятный TypeScript
- Секреты читай только через Deno.env.get() — никогда не хардкодь токены
- Не меняй схему БД без явной команды
- Не трогай файлы, которые не относятся к задаче
- Соблюдай инварианты домена (например, Economy — единственный модуль для работы с балансом)

Перед реализацией:
1. Опиши, что ты понял из задачи
2. Перечисли файлы, которые будешь создавать или менять
3. Покажи план действий
4. Только потом пиши код

Если чего-то не хватает (нет описания API, нет контракта) — задай уточняющий вопрос.
```

## 4. Шаблон для изменения документации (docs-as-code)

**Когда использовать:** При обновлении документации после изменений в бизнес-логике, API-контрактах или архитектуре.

```
Контекст: проект gamify-artifacts, документация в папке docs/ — источник правды об архитектуре и контрактах.

Произошли изменения в коде, которые нужно отразить в документации.

Изменения в коде: [ОПИШИ, ЧТО ИЗМЕНИЛОСЬ]

Файлы документации, которые нужно обновить:
- docs/overview.md — если изменились цели или идея проекта
- docs/domain-model.md — если изменились доменные модули или общая архитектура
- docs/ui-screens.md — если изменились экраны или UI
- docs/integration/singularity.md — если изменилась интеграция с Singularity
- docs/api/ — если изменились контракты Edge Functions
- docs/ai-guidelines.md — если изменились правила работы

Требования к обновлению документации:
- Документация должна оставаться актуальной и быть источником правды
- Описывай изменения чётко и конкретно, без воды
- Сохраняй структуру и стиль существующих файлов
- Если нужно добавить новый файл в docs/ — создай его
- Если изменение касается API — обнови спецификацию в docs/api/

Перед внесением изменений:
1. Прочитай текущее содержимое файлов документации, которые будешь менять
2. Опиши, что именно ты собираешься изменить и почему
3. Только потом вноси правки
```

## 5. Шаблон для «разобраться в существующем коде перед изменениями»

**Когда использовать:** Перед нетривиальными изменениями, когда нужно понять текущую реализацию, найти точки расширения или выявить зависимости.

```
Контекст: проект gamify-artifacts, модульный монолит (Supabase + React + Vite).

Мне нужно разобраться в существующем коде перед тем, как вносить изменения.

Задача, которую я хочу решить: [ОПИШИ ЗАДАЧУ]
Домен, которого это касается: [Player / Economy / Inventory / Shop / Artifacts / SingularityIntegration]

Перед тем как что-либо менять, сделай следующее:

1. Прочитай соответствующие файлы в docs/:
   - docs/ai-guidelines.md (правила)
   - docs/domain-model.md (доменные модули)
   - docs/domain-model.md (доменные модули и архитектура)
   - Другие релевантные файлы из docs/

2. Найди и прочитай весь существующий код, который относится к этой задаче:
   - Edge Functions (backend)
   - React-компоненты и стейт (фронтенд)
   - Типы и интерфейсы
   - Тесты, если есть

3. Опиши мне:
   - Как сейчас реализована логика, которую я хочу изменить
   - Какие файлы и модули задействованы
   - Какие зависимости и связи между модулями существуют
   - Где точки расширения — что можно добавить, а что лучше не трогать
   - Есть ли нарушения инвариантов или потенциальные проблемы

4. Предложи план изменений:
   - Какие файлы нужно создать или изменить
   - В каком порядке лучше действовать
   - Что может сломаться и как это проверить

НЕ пиши код на этом этапе — только анализ и план.
Если чего-то не хватает или непонятно — задай уточняющий вопрос.
```
