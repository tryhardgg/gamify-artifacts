# Security & Secrets Guidelines

## 1. Общие принципы

- Репозиторий `gamify-artifacts` публичный, поэтому **любые реальные секреты и токены не должны попадать в Git**.  
- Все чувствительные значения хранятся либо:
  - в `.env.local` / `.env` локально (и игнорируются Git),  
  - либо в **Supabase secrets** (project configuration) на стороне бэкенда.  
- Если секрет случайно закоммичен — он считается скомпрометированным и должен быть немедленно отозван/перегенерирован.

## 2. Классификация ключей

### 2.1. Публичные ключи (можно использовать во фронте)

- `VITE_SUPABASE_URL`  
- `VITE_SUPABASE_ANON_KEY`

Эти ключи предназначены для использования в браузере и по дизайну Supabase могут быть видны на фронте.  
Тем не менее, они не должны давать доступ к операциям, которые обходят Row Level Security.

### 2.2. Приватные ключи (нельзя светить)

- Любые сервис‑ключи Supabase (service role key).  
- Токен доступа к Singularity API.  
- Любые другие API‑ключи внешних сервисов.

Эти ключи **никогда не должны находиться в репозитории или в клиентском коде**.

## 3. Фронтенд (GitHub Pages + Vite)

- Файл `.env.local` используется для локальной разработки и **игнорируется Git**.  
- Для билда фронта на GitHub Actions (если будет использован CI):
  - Секреты (URL и anon key Supabase) задаются в GitHub → Settings → Secrets.  
  - В workflow они прокидываются как переменные окружения и используются на этапе `npm run build`.  

- В исходном коде:
  - Нельзя хардкодить ключи типа `https://xxx.supabase.co` и `anon_key` прямо в файлах.  
  - Использовать только `import.meta.env.VITE_*` переменные.

## 4. Backend (Supabase Edge Functions)

- Все приватные ключи и токены хранятся в **Supabase secrets** (Project Settings → Configuration / Secrets).  
  Примеры:
  - `SINGULARITY_API_TOKEN`  
  - `SUPABASE_SERVICE_ROLE_KEY` (если потребуется)

- Edge Functions читают секреты только через переменные окружения:

```ts
const singularityToken = Deno.env.get('SINGULARITY_API_TOKEN');
```

- В коде **запрещено**:
  - хардкодить токены и пароли;  
  - логировать значение секретов;  
  - отдавать секреты во фронт в ответах API.

## 5. Проверки перед коммитом

Перед пушем в публичный репозиторий желательно:

- Выполнять поиск по ключевым словам: `token`, `api_key`, `supabase`, `password`.  
- Убедиться, что файлы `.env`, `.env.local` и любые файлы с секретами присутствуют в `.gitignore`.  
- Если обнаружен секрет в истории:
  - немедленно **отозвать/перегенерировать** его в Supabase или Singularity;  
  - удалить секрет из кода и истории (по возможности),  
  - задокументировать инцидент (кратко в issue/notes).

## 6. Особые моменты для Singularity интеграции

- Токен Singularity:
  - хранится только в Supabase secrets (`SINGULARITY_API_TOKEN`);  
  - используется исключительно в серверной части (Edge Function `sync_singularity_tasks`).  
- Фронтенд никогда не обращается напрямую к Singularity API.  
- Любые дополнительные параметры интеграции (ID рабочего пространства, фильтры) по возможности также держать в конфиге/секретах, а не в коде.
